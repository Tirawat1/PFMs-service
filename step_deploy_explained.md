# step_deploy.md — ฉบับอธิบายละเอียด

เอกสารนี้อธิบาย "ทำไม" และ "แต่ละเครื่องมือคืออะไร" ประกอบ [step_deploy.md](step_deploy.md)
ถ้า step_deploy.md คือสูตรอาหาร อันนี้คือ "ทำไมต้องใส่ขั้นตอนนี้"

---

## ภาพรวม: ใครทำหน้าที่อะไร

```
Terraform          →  สร้าง "เครื่อง" (EC2) + เปิดพอร์ตที่ต้องใช้        (รันตอนตั้งค่าครั้งแรก / ไม่บ่อย)
DuckDNS            →  แปลงชื่อ (pfms-yourname.duckdns.org) → IP ของเครื่อง
nginx + certbot     →  เครื่องรับ request จากอินเทอร์เน็ตที่ port 80/443 แล้วส่งต่อเข้า container, ทำ HTTPS ให้
Docker + Compose   →  รันตัวแอปจริง (Next.js) + ฐานข้อมูล (Postgres) เป็น container แยกกัน
GitHub Actions     →  ทุกครั้งที่ push โค้ดใหม่ → SSH เข้าเครื่อง → ดึงโค้ดใหม่ → build ใหม่ → รันใหม่ (อัตโนมัติ, ทุกครั้งที่ push)
```

จุดสำคัญ: **Terraform สร้างเครื่องเปล่า ๆ ที่ยังไม่มีแอปอยู่ข้างใน** ส่วน **GitHub Actions เป็นคนเอาแอปเข้าไปใส่**
สองอย่างนี้ทำงานคนละจังหวะกัน ไม่ใช่ตัวเลือกที่ใช้แทนกันได้ — มันคือ 2 ชั้นที่ต้องมีทั้งคู่

---

## 1. Terraform คืออะไร ทำไมต้องใช้

Terraform เป็นเครื่องมือ **Infrastructure as Code (IaC)** — แทนที่จะเข้าไปคลิกสร้าง EC2 ใน AWS Console ด้วยมือ (คลิกเลือก AMI, คลิกตั้ง security group, คลิก allocate Elastic IP ทีละอัน) เราเขียนเป็นไฟล์ `.tf` แทน แล้วสั่ง `terraform apply` ครั้งเดียวมันสร้างให้ครบ

**ข้อดีที่สำคัญ:**
- **Reproducible** — เครื่องพังหรือ region ล่ม สร้างใหม่ด้วยคำสั่งเดิมได้ทันที ไม่ต้องจำว่าคลิกอะไรไปบ้าง
- **Version control ได้** — ไฟล์ `.tf` อยู่ใน git เหมือนโค้ด เห็น diff ได้ว่าใครเปลี่ยน infra อะไรเมื่อไหร่
- **ลบทิ้งสะอาด** — `terraform destroy` ลบทุกอย่างที่มันสร้าง ไม่มีของค้าง (ต่างจากคลิกลบเองที่มักลืมบางอย่าง เช่น Elastic IP ที่ยังคิดเงินอยู่)

**สิ่งที่ terraform/ ในโปรเจกต์นี้สร้าง (ดู [terraform/main.tf](terraform/main.tf)):**
- `aws_instance` — ตัวเครื่อง EC2 (Ubuntu 22.04)
- `aws_security_group` — "กำแพงไฟร์วอลล์" กำหนดว่าพอร์ตไหนเปิดรับจากใครได้บ้าง (22=SSH, 80/443=web)
- `aws_eip` — Elastic IP คือ IP แบบตายตัว ถ้าไม่มีอันนี้ ทุกครั้งที่ restart เครื่อง IP จะเปลี่ยน แล้ว DuckDNS ที่ชี้ไว้จะพังทันที
- `user_data.sh` — สคริปต์ที่รันอัตโนมัติตอนเครื่องเปิดครั้งแรก (ติดตั้ง Docker, nginx, certbot ให้)

**สิ่งที่ Terraform ไม่ทำ:** ไม่รู้จักโค้ด Next.js ของเราเลย ไม่ build ไม่ deploy แอป — มันแค่เตรียม "บ้านเปล่า" ที่มีปลั๊กไฟและประปาพร้อมใช้

**ครั้งแรกที่ใช้ terraform apply จะเกิดอะไรขึ้น:**
1. อ่านไฟล์ `.tf` ทั้งหมดในโฟลเดอร์ แปลงเป็นแผนการเปลี่ยนแปลง (`terraform plan` ทำแบบนี้ให้ดูก่อนได้)
2. เรียก AWS API สร้างทุกอย่างตามแผน
3. บันทึกสถานะ ("ตอนนี้มีอะไรอยู่บ้าง") ลงไฟล์ `terraform.tfstate` ในเครื่องคุณ — **ไฟล์นี้สำคัญมาก อย่า commit เข้า git แล้วอย่าลบทิ้ง** เพราะครั้งต่อไปที่รัน `terraform apply` หรือ `destroy` มันใช้ไฟล์นี้เทียบว่าอะไรมีอยู่แล้วบ้าง ถ้าไฟล์หายมันจะไม่รู้จักเครื่องที่สร้างไปแล้ว (แล้วอาจพยายามสร้างซ้ำ)

> เก็บ `terraform.tfstate` ไว้ในเครื่อง หรือถ้าจะทำงานเป็นทีม/หลายเครื่อง ค่อยศึกษาเรื่อง "remote state" (เช่นเก็บใน S3) ทีหลัง — ตอนนี้ใช้คนเดียว เก็บในเครื่องพอ

## 2. GitHub Actions คืออะไร ต่างจาก Terraform ยังไง (ตอบคำถามที่ถามมา)

GitHub Actions คือ **CI/CD** — ระบบที่รันสคริปต์อัตโนมัติเมื่อมี "เหตุการณ์" เกิดขึ้นในรีโป (เช่น push โค้ด, เปิด PR) มันรันบนเครื่องของ GitHub เอง (cloud runner ที่ GitHub จัดให้ฟรีตามโควตา) ไม่ใช่รันบน EC2 ของเรา

ในไฟล์ [.github/workflows/deploy.yml](.github/workflows/deploy.yml) ที่สร้างไว้:
```
push โค้ดไป main
  → GitHub สร้างเครื่องชั่วคราว (runner)
  → runner นั้น SSH เข้า EC2 ของเรา (ใช้ private key ที่เก็บเป็น secret)
  → สั่งคำสั่งบน EC2: git pull, docker compose build, docker compose up -d
  → runner ปิดตัวเอง (ใช้ครั้งเดียวทิ้ง)
```

**เทียบให้เห็นภาพ:**

| | Terraform | GitHub Actions |
|---|---|---|
| สร้างอะไร | เครื่อง (EC2), เครือข่าย, IP | เอาโค้ดล่าสุดขึ้นไปรันบนเครื่องที่มีอยู่แล้ว |
| รันตอนไหน | มือเรากดเอง เวลาจะเปลี่ยน infra (นาน ๆ ครั้ง) | อัตโนมัติทุกครั้งที่ push (บ่อย) |
| รันที่ไหน | เครื่องเราเอง (หรือ CI) เรียก AWS API | เครื่อง GitHub ชั่วคราว แล้ว SSH เข้า EC2 |
| ถ้าไม่มีอันนี้ | จะไม่มีเครื่องให้ deploy เข้าไปเลย | ก็ยังต้อง SSH เข้าไป deploy มือทุกครั้งที่แก้โค้ด |

**ทำไมต้องมีทั้งคู่:** GitHub Actions ต้องมี "เครื่องปลายทาง" อยู่ก่อนถึงจะ SSH เข้าไปได้ — เครื่องนั้นมาจาก Terraform ขั้นตอนที่ 1 เสมอ เรียงลำดับคือ **Terraform ก่อน (สร้างบ้าน) → GitHub Actions ทีหลัง (ส่งของเข้าบ้านทุกครั้งที่มีของใหม่)**

ถ้าวันหนึ่งอยากรื้อเครื่องทิ้งแล้วสร้างใหม่ (เช่นย้าย region, อัปเกรด instance type) — แก้ที่ Terraform แล้ว `apply` ใหม่ ไม่เกี่ยวกับ GitHub Actions เลย GitHub Actions จะยังทำงานเหมือนเดิมกับเครื่องใหม่ (แค่ต้องอัปเดต secret `EC2_HOST` เป็น IP ใหม่)

## 3. Docker / docker-compose คืออะไร

- **Docker image** = พิมพ์เขียวของแอป (โค้ด + runtime + dependencies ทั้งหมด) แพ็กเป็นไฟล์เดียว รันที่ไหนก็ได้ผลเหมือนกัน ไม่มีปัญหา "ในเครื่องผมมันรันได้นะ"
- **Docker container** = เอา image มา "รัน" จริง กลายเป็น process ที่แยก (isolate) จากเครื่องแม่และ container อื่น
- **docker-compose.yml** = ไฟล์บอกว่าต้องรันกี่ container พร้อมกัน เชื่อมกันยังไง — ในโปรเจกต์นี้มี 2 container: `app` (Next.js) กับ `db` (Postgres) มันคุยกันผ่าน network ภายในของ Docker เอง โดยที่ `db` **ไม่เปิดพอร์ตออกสู่อินเทอร์เน็ตเลย** (ดู [docker-compose.yml](docker-compose.yml) — ไม่มี `ports:` ใต้ service `db`) ปลอดภัยกว่าเปิดฐานข้อมูลออกสาธารณะ

**ทำไมไม่ใช้ RDS ตอนนี้:** RDS คือ Postgres แบบ managed service ของ AWS (AWS ดูแล backup/patch/scaling ให้) สะดวกกว่าแต่มีค่าใช้จ่ายเพิ่ม การรัน Postgres เป็น container ข้าง ๆ แอปบน EC2 เดียวกัน (ตามที่ทำอยู่นี้) ประหยัดกว่าสำหรับเริ่มต้น/ทดลอง แต่ต้อง backup เอง (ดูหัวข้อ 6 ใน step_deploy.md) — ถ้าจะย้ายไป RDS ทีหลังทำได้โดยไม่ต้องแก้โค้ดแอปเลย แค่เปลี่ยนค่า `DATABASE_URL`

## 4. nginx + certbot คืออะไร ทำไมต้องมี

Next.js container เปิดพอร์ต 3000 ไว้ แต่เรา **ไม่เปิดพอร์ต 3000 ออกอินเทอร์เน็ตตรง ๆ** ([docker-compose.yml](docker-compose.yml) bind ไว้แค่ `127.0.0.1:3000` คือรับได้เฉพาะจากในเครื่องเดียวกัน)

แทนที่จะเปิดตรง เราให้ **nginx** (โปรแกรมที่ลงอยู่บนตัว EC2 host เอง ไม่ใช่ container) เป็นคนเดียวที่ฟังพอร์ต 80/443 จากอินเทอร์เน็ต แล้ว "ส่งต่อ" (reverse proxy) เข้าไปที่ 127.0.0.1:3000 อีกที เหตุผล:
- รวมจุดเข้าออกไว้ที่เดียว จัดการ TLS/certificate ได้ในที่เดียว
- ถ้าจะเพิ่มบริการอื่นทีหลัง (เช่นอีก container หนึ่ง) เพิ่ม route ใน nginx ได้โดยไม่ต้องเปลี่ยนพอร์ตที่เปิดสู่โลกภายนอก

**certbot** คือตัวขอใบรับรอง SSL/TLS ฟรีจาก Let's Encrypt ให้อัตโนมัติ คำสั่ง `certbot --nginx` จะ:
1. พิสูจน์ว่าเราเป็นเจ้าของโดเมนจริง (เช็คว่า DNS ของ `pfms-yourname.duckdns.org` ชี้มาที่เครื่องนี้จริง)
2. ออกใบรับรองให้
3. **แก้ไฟล์ config ของ nginx ให้เองอัตโนมัติ** เพิ่ม server block สำหรับ HTTPS (port 443) + ตั้ง redirect จาก http → https
4. ตั้ง renewal timer อัตโนมัติ (ใบรับรองมีอายุ 90 วัน ต่ออายุให้เองโดยไม่ต้องทำอะไร)

## 5. DuckDNS คืออะไร

โดเมนปกติ (เช่น `.com`) ต้องซื้อ DuckDNS คือบริการ **ฟรี** ที่ให้ subdomain (`ชื่อที่ตั้ง.duckdns.org`) แล้วให้เราสั่งอัปเดตว่ามันชี้ไป IP ไหนผ่าน API call ง่าย ๆ (ดู step 2 ใน step_deploy.md) เหมาะกับ "ลองของ" ก่อนจะซื้อโดเมนจริง — HTTPS/certbot ใช้ได้ปกติกับ DuckDNS เพราะ Let's Encrypt ไม่สนว่าโดเมนมาจากไหน แค่พิสูจน์ความเป็นเจ้าของได้ก็พอ

**ข้อจำกัด:** ถ้าเครื่อง EC2 ไม่มี Elastic IP (ใช้ IP ปกติที่เปลี่ยนทุกครั้ง restart) ต้องมี cron คอยยิง DuckDNS update ทุกครั้งที่ IP เปลี่ยน — แต่ในเซ็ตอัปนี้เรามี Elastic IP แล้ว (จาก Terraform) ที่ตายตัว เลยตั้งครั้งเดียวจบ ไม่ต้องมี cron

## สรุปลำดับการรันจริง (ครั้งแรก)

```
1. terraform apply         → ได้เครื่อง + IP ตายตัว
2. DuckDNS update          → ชื่อโดเมนชี้ไปที่เครื่อง
3. SSH เข้าไป clone repo, ตั้ง .env, docker compose up   → แอปรันอยู่ที่ 127.0.0.1:3000 ในเครื่อง
4. ตั้ง nginx + certbot     → โดเมนเข้าถึงได้จากอินเทอร์เน็ตแบบ HTTPS
5. ตั้ง GitHub secrets      → จากนี้ push ครั้งต่อไป deploy อัตโนมัติ ไม่ต้อง SSH มือเองอีก
```

ขั้นตอน 1-4 ทำ "ครั้งเดียว" ตอนตั้งเครื่องใหม่ ส่วนขั้นตอน 5 (GitHub Actions) จะทำงาน "ทุกครั้ง" หลังจากนั้นโดยอัตโนมัติ
